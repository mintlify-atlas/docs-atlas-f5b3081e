## Overview

KoroFood uses **Netflix Eureka** as its service discovery solution. Eureka enables microservices to:

- **Register themselves** on startup
- **Discover other services** dynamically
- **Monitor service health** with heartbeats
- **Enable load balancing** without hardcoded addresses

<Info>
The Eureka Server runs on port **8761** and serves as the central service registry for all microservices.
</Info>

## Eureka Server Setup

### Main Application Class

From `eurekaService/src/main/java/com/koroFoods/eurekaService/EurekaServiceApplication.java`:

```java
package com.koroFoods.eurekaService;

import org.springframework.boot.SpringApplication;
import org.springframework.boot.autoconfigure.SpringBootApplication;
import org.springframework.cloud.netflix.eureka.server.EnableEurekaServer;

@SpringBootApplication
@EnableEurekaServer
public class EurekaServiceApplication {

    public static void main(String[] args) {
        SpringApplication.run(EurekaServiceApplication.class, args);
    }
}
```

<Note>
The `@EnableEurekaServer` annotation is all you need to enable Eureka Server functionality.
</Note>

### Configuration

From `eurekaService/src/main/resources/application.properties`:

```properties
spring.application.name=eurekaService
server.port=8761

# Standalone mode - don't register with itself
eureka.client.register-with-eureka=false
eureka.client.fetch-registry=false

# Health endpoint exposure
management.endpoints.web.exposure.include=health
management.endpoint.health.show-details=always
```

### Key Configuration Properties

<AccordionGroup>
  <Accordion title="eureka.client.register-with-eureka=false">
    Prevents the Eureka Server from trying to register itself as a client. In standalone mode, the server doesn't need to register with itself.
  </Accordion>
  
  <Accordion title="eureka.client.fetch-registry=false">
    Prevents the server from trying to fetch the service registry. Since it IS the registry, it doesn't need to fetch from anywhere.
  </Accordion>
  
  <Accordion title="management.endpoints.web.exposure.include=health">
    Exposes the `/actuator/health` endpoint for health checks and monitoring.
  </Accordion>
</AccordionGroup>

## Docker Configuration

From `docker-compose.yml`:

```yaml
eureka-server:
  build: ./eurekaService
  ports:
    - "8761:8761"
  healthcheck:
    test: ["CMD-SHELL", "wget -qO- http://localhost:8761/actuator/health || exit 1"]
    interval: 15s
    timeout: 10s
    retries: 10
    start_period: 60s
```

<Warning>
The `start_period` of 60 seconds allows Eureka Server adequate time to initialize before health checks begin.
</Warning>

## Service Registration

### Client Configuration

All microservices register with Eureka using this configuration pattern:

```yaml
# Example: User Service
user-service:
  build: ./userService/userService
  ports:
    - "8081:8081"
  depends_on:
    eureka-server:
      condition: service_healthy
  environment:
    EUREKA_CLIENT_SERVICEURL_DEFAULTZONE: http://eureka-server:8761/eureka/
    EUREKA_INSTANCE_PREFER_IP_ADDRESS: "true"
```

### Spring Configuration

Services include these properties in their `application.properties`:

```properties
spring.application.name=userService

# Eureka client configuration
eureka.client.service-url.defaultZone=http://localhost:8761/eureka/
eureka.client.register-with-eureka=true
eureka.client.fetch-registry=true
eureka.instance.prefer-ip-address=true
```

### Registration Flow

<Steps>
  <Step title="Service Startup">
    Microservice starts and loads Eureka client configuration
  </Step>
  <Step title="Registration Request">
    Service sends registration request to Eureka Server with metadata:
    - Service name (e.g., "userService")
    - IP address or hostname
    - Port number
    - Health check URL
  </Step>
  <Step title="Heartbeat Initiation">
    Service begins sending heartbeats every 30 seconds
  </Step>
  <Step title="Registry Replication">
    Eureka Server updates its registry and notifies other clients
  </Step>
  <Step title="Service Ready">
    Service is now discoverable by other microservices
  </Step>
</Steps>

## Service Discovery

### How Services Find Each Other

Services use Eureka to discover other services dynamically:

```java
// Example: Using Feign Client with Eureka
@FeignClient(name = "userService")
public interface UserServiceClient {
    @GetMapping("/user/feign/{id}")
    UserDTO getUser(@PathVariable Long id);
}
```

<Info>
The `name = "userService"` parameter matches the `spring.application.name` of the target service in Eureka.
</Info>

### Discovery Process

<Steps>
  <Step title="Client Request">
    Service needs to call another service (e.g., Order Service calls User Service)
  </Step>
  <Step title="Registry Query">
    Client queries local cache of Eureka registry for "userService"
  </Step>
  <Step title="Instance Selection">
    If multiple instances exist, load balancer selects one
  </Step>
  <Step title="HTTP Request">
    Client makes HTTP request to selected instance's IP and port
  </Step>
  <Step title="Cache Update">
    Client periodically refreshes registry cache (every 30 seconds)
  </Step>
</Steps>

## Health Checks

### Service Health Monitoring

Eureka monitors service health through heartbeats:

```yaml
# Service health check configuration
healthcheck:
  test: ["CMD-SHELL", "wget -qO- http://localhost:8081/actuator/health || exit 1"]
  interval: 10s
  timeout: 5s
  retries: 5
```

### Heartbeat Mechanism

<CardGroup cols={2}>
  <Card title="Heartbeat Interval" icon="heartbeat">
    Services send heartbeats every **30 seconds**
  </Card>
  <Card title="Eviction Timeout" icon="clock">
    Services are evicted after **90 seconds** without heartbeat
  </Card>
  <Card title="Self-Preservation" icon="shield">
    Eureka enters self-preservation if >15% services fail
  </Card>
  <Card title="Registry Replication" icon="copy">
    Changes propagate to all clients within **30 seconds**
  </Card>
</CardGroup>

### Service Status States

- **UP**: Service is healthy and accepting traffic
- **DOWN**: Service failed health check
- **STARTING**: Service is initializing
- **OUT_OF_SERVICE**: Service is temporarily unavailable
- **UNKNOWN**: Health status cannot be determined

## Load Balancing

### Client-Side Load Balancing

Eureka integrates with **Spring Cloud LoadBalancer** for client-side load balancing:

```java
// Spring Cloud Gateway automatically load balances
.route("user-service",
    r -> r.path("/auth/**")
        .uri("lb://userService")) // lb:// prefix enables load balancing
```

### Load Balancing Strategies

<Tabs>
  <Tab title="Round Robin">
    Default strategy. Distributes requests evenly across all instances:
    
    ```
    Request 1 → Instance A
    Request 2 → Instance B
    Request 3 → Instance C
    Request 4 → Instance A
    ```
  </Tab>
  
  <Tab title="Random">
    Randomly selects an instance for each request:
    
    ```java
    @Bean
    public ReactorLoadBalancer<ServiceInstance> randomLoadBalancer(
        Environment environment,
        LoadBalancerClientFactory loadBalancerClientFactory) {
        return new RandomLoadBalancer(
            loadBalancerClientFactory
                .getLazyProvider(name, ServiceInstanceListSupplier.class),
            name
        );
    }
    ```
  </Tab>
  
  <Tab title="Weighted">
    Distributes traffic based on instance weights (requires custom configuration):
    
    ```
    Instance A (weight: 3) → 60% of traffic
    Instance B (weight: 2) → 40% of traffic
    ```
  </Tab>
</Tabs>

## Scaling Services

### Running Multiple Instances

To run multiple instances of a service:

```bash
# Scale user service to 3 instances
docker-compose up --scale user-service=3
```

Eureka automatically:
1. Registers all 3 instances
2. Load balances requests between them
3. Removes failed instances from rotation

<Warning>
Ensure services are stateless or use shared session storage (like Redis) when scaling horizontally.
</Warning>

## Troubleshooting

### Service Not Appearing in Eureka

<AccordionGroup>
  <Accordion title="Check Eureka Server Health">
    ```bash
    curl http://localhost:8761/actuator/health
    ```
    
    Expected response:
    ```json
    {"status":"UP"}
    ```
  </Accordion>
  
  <Accordion title="Verify Service Configuration">
    Ensure service has correct Eureka client configuration:
    
    ```properties
    eureka.client.service-url.defaultZone=http://eureka-server:8761/eureka/
    eureka.client.register-with-eureka=true
    eureka.client.fetch-registry=true
    ```
  </Accordion>
  
  <Accordion title="Check Network Connectivity">
    ```bash
    # From service container
    docker exec <service-container> ping eureka-server
    ```
  </Accordion>
  
  <Accordion title="Review Service Logs">
    ```bash
    docker logs <service-container> | grep -i eureka
    ```
    
    Look for registration confirmation:
    ```
    DiscoveryClient_USER-SERVICE/192.168.1.5:user-service:8081 - registration status: 204
    ```
  </Accordion>
</AccordionGroup>

### Service Discovery Failures

<Steps>
  <Step title="Verify Service Name">
    Ensure Feign client name matches exact service name in Eureka:
    ```java
    @FeignClient(name = "userService") // Must match spring.application.name
    ```
  </Step>
  
  <Step title="Check Registry Cache">
    Registry updates can take up to 30 seconds. Wait and retry.
  </Step>
  
  <Step title="Force Registry Refresh">
    Restart the calling service to force registry refresh.
  </Step>
</Steps>
